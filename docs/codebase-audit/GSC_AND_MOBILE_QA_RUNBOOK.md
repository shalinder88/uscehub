## Part 1 — Search Console domain property + sitemap submission

### Why now

The verified-USCE wedge is now honest end-to-end on production:

- PR #11 fixed the 405 audit semantics
- PR #13 made listing-detail trust language honest
- PR #16 made cards / compare / recommend honest
- PR #17 made the grid order match the trust language

That makes this the right moment to ask Google to crawl. Before this, indexing the site at scale would have promoted the green "Verified link" overclaim across 136 legacy-backfilled rows. Now the public language is conservative.

### Pre-flight (read-only)

Confirm production canonical alignment is unchanged:

```bash
curl -sI https://uscehub.com/         # expect 200
curl -sI https://www.uscehub.com/     # expect 308 → https://uscehub.com/
curl -sI https://uscehub.com/sitemap.xml | head -5
```

The third probe should return `200` and serve the dynamically generated sitemap. The sitemap excludes any listing the route logic deems non-canonical (PR 3.5 / 3.5b / 3.7 do not change which listings are in the sitemap — they only change badge labels and ordering).

### Steps

1. **Add the domain property** at [search.google.com/search-console](https://search.google.com/search-console).
   - Pick **Domain** (not URL prefix). This covers apex + www + http + https in a single property.
   - Property: `uscehub.com`.

2. **DNS-verify**:
   - Search Console gives you a TXT record value like `google-site-verification=<token>`.
   - Add a TXT record on the apex of `uscehub.com` (whichever DNS provider owns the zone — Cloudflare, Vercel-managed DNS, or Namecheap).
   - Wait for DNS propagation (usually ≤ 5 minutes; can be up to an hour). Click "Verify" in Search Console.

3. **Submit the sitemap**:
   - Sitemaps → Add a new sitemap.
   - Enter `https://uscehub.com/sitemap.xml` (relative path `sitemap.xml` is also accepted in the Domain property).
   - Confirm Search Console reports "Success" within ~24 hours. The first crawl can take days.

4. **Sanity-watch the Coverage report** for the first week:
   - **Indexed** count should grow over a few days.
   - **Excluded — Crawled but not currently indexed** is normal for low-traffic listing pages; not a blocker.
   - **Errors** — investigate any `Server error (5xx)`, `Redirect error`, or `Submitted URL marked noindex` immediately. None of those should appear on a healthy crawl.

### What this does NOT change

- No code change. Sitemap is already generated by [src/app/sitemap.xml/route.ts](https://github.com/shalinder88/uscehub) (or equivalent — it's a Next.js dynamic route).
- No `robots.txt` change.
- No canonical metadata change.

### When to skip

If the team decides to delay public crawling (e.g. pending more cron ticks before Google fully indexes), simply do not submit the sitemap. Domain verification alone is harmless — it is a one-time TXT record.

## Part 2 — Mobile QA checklist for the trust UI

### Why now

PR #13, #16, and #17 changed what users see on:

- listing detail trust block
- listing card badges (browse, observerships, featured)
- compare table verification cell
- listing grid order

Desktop QA was sufficient to ship each PR (Vercel preview deploys, code review). **Mobile was not specifically QAed.** Given how much of USCEHub's primary audience (IMG applicants, residents on call) reads on phones, real-device verification is overdue.

### Test on real devices, not devtools

Devtools mobile emulation lies about:

- touch-target hit areas
- font rendering at low DPI
- thumb-reachability
- system theme (dark mode trips on iOS sooner than Chrome devtools shows)
- right-edge clipping inside Safari iOS chrome bar

Pick **one iOS** (any iPhone running current Safari) and **one Android** (any modern Chrome).

### Pages to walk

For each page, confirm: no horizontal scroll, no clipped tap targets, no overlapping text, no badge wraps mid-word, dark mode legible.

1. `https://uscehub.com/` — homepage. Confirm hero, featured listings grid, navigation tap targets.
2. `https://uscehub.com/browse` — full listing grid. Confirm cards render, badges visible, filters scrollable, sort menu opens.
3. `https://uscehub.com/observerships` — state index.
4. `https://uscehub.com/observerships/CA` (or similar) — state-level grid.
5. `https://uscehub.com/observerships/specialty/internal-medicine` — specialty-level grid.
6. **One verified listing detail** — a listing whose `linkVerificationStatus = VERIFIED` AND `lastVerifiedAt` is set. (After tomorrow's cron tick: e.g. Indiana University Health, Jamaica Hospital from the 2026-04-28 manual run.) Confirm:
   - Green "Verified link" badge
   - "Last verified [relative time]" line below the badge
   - "Apply Now" button
   - "Report broken link" button still visible
   - `<ListingDisclaimer>` still visible at the top
7. **One legacy verified-on-file listing detail** — Stanford Health Care or BronxCare (`VERIFIED + lastVerifiedAt = null`). Confirm:
   - Slate "Official source on file" badge — **NOT green**
   - No "Last verified ..." line (no fake date)
   - "Apply Now" still works (URL is on file)
8. **One needs-review listing detail** — UPMC or Hackensack (`NEEDS_MANUAL_REVIEW` from the cron run). Confirm:
   - Amber "Source needs review" badge with AlertTriangle icon
   - "Source needs review — verify directly with the institution before applying" subline
   - CTA is "View Official Source", not "Apply Now"
9. **One unknown listing detail** — Mount Sinai (or any `UNKNOWN` listing). Confirm:
   - Soft amber "Source not yet verified" badge
   - No date subline
   - CTA is "View Official Source"
10. `https://uscehub.com/compare` — pick 2–3 listings of mixed states (verified, verified-on-file, needs-review). Confirm the "Verified" row shows the right badge per row, no overflow on small screens, mobile-stacked view fits.
11. `https://uscehub.com/admin` (logged in as ADMIN) — Confirm `/admin/verification-queue` link is in the sidebar, page renders, queue sources visible, action buttons fit. **Confirm /admin is NOT publicly accessible** (log out, hit `/admin`, expect redirect to `/auth/signin`).

### Visual rules to verify

- The "Verified link" green pill **only** appears when `lastVerifiedAt` exists. If it appears on a listing that had a null `lastVerifiedAt` 24 hours ago, the cron may have updated it; that is correct. Cross-check with the cron health script:
  ```bash
  cd /Users/shelly/usmle-platform && npx tsx scripts/check-verify-listings-cron.ts
  ```
- The slate "Official source on file" pill should look **clearly different from green "Verified link"** even on a small screen — the green channel is the trust signal, neutral slate is "we have it but haven't checked recently."
- The amber "Source needs review" pill should be **stronger / more visible** than the soft "Source not yet verified" pill (PR #16 deliberately uses a different shade and icon — `AlertTriangle` vs `ShieldAlert`).
- "Apply Now" button is still safe — only appears for `linkVerified=true` OR `linkVerificationStatus=VERIFIED` rows.
- "Report broken link" affordance is still present on detail pages.
- `<ListingDisclaimer>` ("we are re-verifying application links and deadlines...") is still visible.

### Bug-flag rules

If you spot a mobile-only issue:

- **One small PR per issue.** Do not bundle UI fixes with feature PRs (handoff §9 error pattern #8 — "lint warnings persist; do not fix opportunistically without scope authorization" applies to mobile fixes too).
- File the PR with title `fix(mobile): <surface>: <one-line bug>` and a screenshot.
- Test the fix on the same real device that surfaced the bug, not devtools.

### What this does NOT do

- Does not change any code.
- Does not change the verification doctrine.
- Does not commit to a specific QA cadence.

It is a checklist. Run it once after PR #13/#16/#17 land, then again whenever a future PR touches trust UI.
