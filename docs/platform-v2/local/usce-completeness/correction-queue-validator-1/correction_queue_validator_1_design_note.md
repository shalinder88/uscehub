# Correction Queue Cross-Join Validator — Design Note

**Date:** 2026-05-09
**Sprint:** P99-P97-CORRECTION-QUEUE-VALIDATOR-1

---

## What this validator does

`scripts/validate-p99-correction-queue-cross-join.ts` validates the full correction chain over a queue root directory:

```
[ queue item.json ]──┬──[ audit log.jsonl ]
                     │
                     ├── listing_id → active runtime cards
                     ├── listing_id → staged runtime cards
                     ├── listing_id → staged-runtime-report-issue-mapping listing_map.csv
                     ├── listing_id → staged-runtime-report-issue-mapping evidence_join_map.csv
                     ├── report_ref → listing_map.report_ref
                     └── deep field/string scan → forbidden field keys / forbidden tokens / raw IPs / internal evidence paths
```

## Hard-fail rules

- `LISTING_ID_NOT_IN_ANY_RUNTIME` — listing_id not in active or staged runtime.
- `LISTING_NOT_IN_LISTING_MAP` — listing_id has no row in the listing-map CSV.
- `LISTING_NOT_IN_EVIDENCE_MAP` — listing_id has no row in the evidence-join-map CSV.
- `REPORT_REF_MISMATCH` — report_ref does not match the listing-map's report_ref column for the listing_id.
- `RUNTIME_SET_MISMATCH` — runtime_set='active' but the row only resolves in staged (or vice versa).
- `AUDIT_LOG_MISSING` — queue item exists but no `<correction_id>.jsonl` audit file is found.
- `AUDIT_LOG_EMPTY` — audit JSONL exists but has zero events.
- `AUDIT_CORRECTION_ID_MISMATCH` — first audit event's correction_id does not match the queue item.
- `AUDIT_FIRST_ACTION_NOT_INTAKE_RECEIVED` — first audit event's action is not `intake_received`.
- `AUDIT_FIRST_STATUS_TRANSITION_INVALID` — first audit event is not `NONE → RECEIVED`.
- `ORPHAN_AUDIT_LOG` — audit JSONL exists with no matching queue item.
- `INVALID_STATUS` / `INVALID_PRIORITY` / `INVALID_ISSUE_TYPE` / `INVALID_RUNTIME_SET` / `INVALID_REPORT_REF` — enum violations.
- `MISSING_SOURCE_CONTEXT` / `MISSING_EVIDENCE_JOIN_KEY` — required field missing.
- `LISTING_ID_MISSING` — listing_id is empty AND report_ref is not `pilot-feedback`.
- `FORBIDDEN_TOKEN_IN_QUEUE_ITEM` — `PUBLIC_NOW` / `IMPORT_READY` / `BRIDGE_READY_TO_RUNTIME` / `APPROVED_FOR_PUBLICATION` appears anywhere outside `NO_<token>` safe phrasing.
- `FORBIDDEN_FIELD_KEY_IN_QUEUE_ITEM` — any key matching the intake schema's 38-name forbidden_fields list appears at any depth.
- `INTERNAL_EVIDENCE_PATH_LEAK` — `/screenshots/`, `manual-png-landing-1`, or `batch-3-evidence-landing` substring appears in any field.
- `RAW_IP_IN_QUEUE_ITEM` — IPv4 or IPv6 pattern appears in any field.

## Warnings (non-fatal)

- `ROW_MAPS_TO_STAGED_NOT_ACTIVE` — listing_id resolves only in staged runtime (not yet active/public).
- `ACTIVE_5_EVIDENCE_T7_LANE` — active card evidence files live on the T7 lane (B-005 cosmetic).
- `STAGED_RUNTIME_MISSING_OR_EMPTY` — staged runtime missing.

## What this validator does NOT do

- Does NOT mutate any file.
- Does NOT call the network.
- Does NOT touch the active runtime, staged runtime, app routes, or UI.
- Does NOT check Wayback URL liveness.
- Does NOT enforce dual-signoff rules — those activate only at later state transitions and live in the future review CLI.

## Joins against canonical files

The validator resolves these canonical paths relative to the repo root:

| Source | Path |
|--------|------|
| Active runtime | `src/data/usce/public-listings-pilot.generated.json` |
| Staged runtime | `src/data/usce/public-listings-pilot-staged-batch-2.generated.json` |
| Listing map | `docs/platform-v2/local/usce-completeness/staged-runtime-report-issue-mapping-1/staged_runtime_report_issue_mapping_1_listing_map.csv` |
| Evidence join map | `docs/platform-v2/local/usce-completeness/staged-runtime-report-issue-mapping-1/staged_runtime_report_issue_mapping_1_evidence_join_map.csv` |

Active runtime is required; if missing or empty the validator hard-fails. Staged runtime is required for staged-row joins; if missing it warns but does not block validation of active-row queue items.

## Test plan

- `test-output-valid/` — 4 valid fixtures generated by the existing smoke driver: 1 active (Morristown), 2 staged (UPMC + Lincoln), 1 generic feedback (no listing_id).
- `test-output-invalid/` — 4 hand-authored sub-roots, each isolating one expected failure mode:
  - `unknown-listing/` → expect `LISTING_ID_NOT_IN_ANY_RUNTIME` + `LISTING_NOT_IN_LISTING_MAP` + `LISTING_NOT_IN_EVIDENCE_MAP`
  - `report-ref-mismatch/` → expect `REPORT_REF_MISMATCH`
  - `missing-audit/` → expect `AUDIT_LOG_MISSING`
  - `public-now-leak/` → expect `FORBIDDEN_TOKEN_IN_QUEUE_ITEM`

Each sub-root is its own queue root and validated independently.

## Out of scope

- File queue implementation changes (Sprint 1 already shipped).
- Rate limiting (deferred).
- `/contact` UI wiring (deferred).
- Cross-validator joins involving Wayback liveness (deferred).
